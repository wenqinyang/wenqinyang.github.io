# Prometheus 与 Grafana 监控告警

# Prometheus

# 什么是Prometheus？

Prometheus是一个开源系统监控和警报工具。Prometheus与Grafana强强联合可以组成强大的监控工具，用于收集、存储、查询和可视化应用程序和系统的指标数据。它最初由 SoundCloud 开发，并于2012年发布为开源项目。

# 它有什么特点?

  • Prometheus 是一个开源的系统监控和警报工具。
  • 它通过拉取方式定期从各种目标（例如应用程序、数据库、操作系统）收集指标数据。
  • Prometheus 提供了强大的查询语言 PromQL 来执行实时数据查询和聚合操作。
  • 它还支持警报规则，可以在特定条件满足时触发警报通知。

# 什么指标(metrics)?

metrics可以理解为：数字测量。时间序列意味着随着时间的推移记录变化。用户想要测量的内容因应用程序而异。对于 Web 服务器来说，它可能是请求时间，对于数据库来说，它可能是活动连接数或活动查询数等。指标在理解应用程序为何以某种方式运行方面发挥着重要作用。例如，当请求数量较多时，应用程序可能会变慢。如果您有请求计数指标，您可以找出原因并增加服务器数量来处理负载。

# 我们为什么要用它而不是别的？

  • 简单易用：配置和部署相对简单，拥有用户友好的界面和直观的查询语言。
  • 多维度数据模型：使用基于标签的多维度数据模型来存储和查询时间序列数据。
  • 强大的查询语言：PromQL 作为 Prometheus 的查询语言，具有丰富的功能和强大的表达能力。
  • 强大的告警系统：灵活而可靠的告警系统，能够根据自定义的规则检测和触发告警。

# 什么时候适合用？

Prometheus works well for recording any purely numeric time series. It fits both machine-centric monitoring as well as monitoring of highly dynamic service-oriented architectures. In a world of microservices, its support for multi-dimensional data collection and querying is a particular strength.

Prometheus 非常适合记录任何纯数字时间序列。它既适合以机器为中心的监控，也适合高度动态的面向服务的架构的监控。在微服务的世界中，它对多维数据收集和查询的支持是一个特殊的优势。

# 什么时候不适合用？

Prometheus values reliability. You can always view what statistics are available about your system, even under failure conditions. If you need 100% accuracy, such as for per-request billing, Prometheus is not a good choice as the collected data will likely not be detailed and complete enough.

如果您需要 100% 的准确性，例如按请求计费，Prometheus 不是一个好的选择，因为收集的数据可能不够详细和完整。

# Prometheus的重要组件

# Prometheus Server(重要)

Prometheus Server 是整个架构中的核心组件，它负责数据采集、存储、查询和报警等功能。Prometheus Server 定期从配置的目标（通常是被监控的应用程序）中获取指标数据，并将其存储在本地的时序数据库中。同时，Prometheus Server 还提供了灵活的查询语言 PromQL，用于对指标数据进行查询和分析。

# Alertmanager(重要)

Alertmanager 负责处理来自 Prometheus Server 的报警规则产生的报警通知。它支持对报警进行静默、去重、分组和路由等操作，并可以通过电子邮件、PagerDuty、Slack 等方式发送报警通知给相应的接收者。

# Exporters(重要)

Exporters 是负责从各种应用程序和系统中采集指标数据的组件。Prometheus 提供了许多官方 Exporters，如 Node Exporter（采集主机级别的指标）、JMX Exporter（采集 Java 应用程序的指标）等。此外，还有一些社区提供的 Exporters，用于采集其他各类应用程序和系统的指标数据。本次我们的示例docker-compose中使用的就是Node Exporter采集器。

# Pushgateway

Pushgateway 允许临时性的指标数据从短期作业或批处理任务中被推送到 Prometheus Server。这在一些无法直接被抓取的情况下很有用，例如批处理任务的指标数据。Pushgateway 可以接收并暂时存储这些指标数据，然后由 Prometheus Server 从 Pushgateway 中获取并处理。

# Grafana

Grafana 是一个开源的数据可视化和仪表板工具，通常与 Prometheus 结合使用。Grafana 可以连接到 Prometheus Server，直接查询和展示 Prometheus 中的指标数据。通过 Grafana，用户可以创建漂亮、交互式的仪表板，用于监控和可视化 Prometheus 中的数据。

# Server与Alertmanager架构关系

在Prometheus中一条告警规则主要由以下几部分组成：

  • 告警名称：用户需要为告警规则命名，当然对于命名而言，需要能够直接表达出该告警的主要内容。
  • 告警规则：告警规则实际上主要由PromQL进行定义，其实际意义是当表达式（PromQL）查询结果持续多长时间（During）后出发告警。

# Alertmanager告警特性

# 分组：

分组机制可以将详细的告警信息合并成一个通知。在某些情况下，比如由于系统宕机导致大量的告警被同时触发，在这种情况下分组机制可以将这些被触发的告警合并为一个告警通知，避免一次性接受大量的告警通知，而无法对问题进行快速定位。

# 抑制：

抑制是指当某一告警发出后，可以停止重复发送由此告警引发的其它告警的机制。

# 静默：

静默提供了一个简单的机制可以快速根据标签对告警进行静默处理。如果接收到的告警符合静默的配置，Alertmanager则不会发送告警通知。

# 路由匹配：

每一个告警都会从配置文件中顶级的route进入路由树，需要注意的是顶级的route必须匹配所有告警(即不能有任何的匹配设置match和match_re)，每一个路由都可以定义自己的接受人以及匹配规则。默认情况下，告警进入到顶级route后会遍历所有的子节点，直到找到最深的匹配route，并将告警发送到该route定义的receiver中。

# PromQL

PromQL 全称：Prometheus Query Language它有以下特点：

  • 简洁易懂：PromQL 使用类似数学表达式的语法，易于理解和编写。它的主要目标是提供一种直观的方式来描述和计算时间序列数据。
  • 时间序列选择：PromQL 允许根据指标名称和标签选择特定的时间序列。您可以使用花括号 {} 来过滤和匹配时间序列。
  • 范围选择：PromQL 允许选择特定时间范围内的数据。您可以使用方括号 [] 和时间表达式来指定起始时间和结束时间。
  • 聚合操作：PromQL 提供了许多聚合操作符，例如求和、平均值、最大值、最小值、计数等，以便对时间序列数据进行聚合计算。这些操作符可以对单个时间序列或多个时间序列进行计算。

# 常用PromQL函数

  1. 计算时间序列的样本数量：count(http_requests_total{status_code="200"}[5m])
  2. 统计特定指标的平均值：avg(metric_name)
  3. 统计特定指标的最大值：max(metric_name)
  4. 统计特定指标的最小值：min(metric_name)
  5. 查询特定时间范围内的指标数据：metric_name[start_time:end_time]
  6. 计算特定指标在一段时间内(5分钟)的速率：rate(metric_name[5m])
  7. 根据标签过滤指标数据：metric_name{label_name="value"}

# PromQL实际运用例子

# 网关业务成功率：

sum(irate(trans_onlinepay_total{status="success"}[5m])) / sum(irate(trans_onlinepay_total{status="all"}[2m])) < 0.5

# 系统QPS波动大于100%：

(sum by(system) (rate(system_request_total[2m])) / sum by(system) (rate(system_request_total[2m] offset 2m)) - 1 > 1) and (sum by(system) (rate(system_request_total[2m])) > 2)

# MQ堆积

rabbitmq_queue_messages{queue=~"base.*"} > 2000

# RedisUserCpuHigh

rate(redis_cpu_user_seconds_total{job="redis_exporter"}[5m]) > 0.4

# 主机cpu使用率

count(http_requests_total{status_code="200"}[5m])

# Grafana

# 什么是Grafana

  • Grafana 是一个开源的可视化平台，用于创建和共享动态仪表盘。
  • 它支持多种数据源，包括 Prometheus、InfluxDB、Elasticsearch 等。
  • Grafana 提供了丰富的图表和面板，可以自定义呈现各种指标数据。
  • 它具有灵活的告警通知机制，可以通过电子邮件、Slack 等方式发送警报通知。

# Grafana的优点是什么？

  • 多数据源支持：支持各种数据来源，如 Prometheus、InfluxDB、Elasticsearch 等。
  • 强大的可视化能力：自定义仪表盘和图表，丰富的可视化选项。
  • 实时监控和报警：通过设置阈值和规则进行告警，并及时通知相关人员。
  • 插件生态系统：丰富的插件库，扩展 Grafana 的功能和数据源支持。
  • 团队协作和共享：轻松共享和协作仪表盘，提高团队的工作效率。

# Grafana多数据源查询

  • 使用 Prometheus 数据源：
  1. 查询 CPU 使用率：
100 - (avg(irate(node_cpu_seconds_total{mode="idle"}[5m])) by (instance) * 100)
  1. 查询网络传入流量：
sum(rate(node_network_receive_bytes_total{job="node"}[5m])) by (instance)
  • 使用 Elasticsearch 数据源：
  1. 查询 HTTP 请求时间：
GET /_search
{
  "query": {
    "bool": {
      "must": [
        { "match": { "status": 200 } }
      ]
    }
  },
  "aggs": {
    "avg_response_time": {
      "avg": { "field": "response_time" }
    }
  }
}
  1. 查询日志关键字统计：
GET /_search
{
  "query": {
    "match": { "message": "error" }
  },
  "aggs": {
    "error_count": {
      "value_count": { "field": "message.keyword" }
    }
  }
}
